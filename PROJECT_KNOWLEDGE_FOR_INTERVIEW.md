# 项目面试知识点整理

这份文档是给你面试前复习用的。目标不是把每个底层原理都背得很深，而是让你能把这个项目的设计思路、技术选型、核心流程和改进点讲清楚。

## 1. 项目一句话介绍

可以这样说：

```text
我做的是一个电商售后智能客服系统，模拟“星选商城”的售后客服“小小易”。用户可以在网页或微信公众号里咨询订单、物流、退款、退货、换货、补发、催发货、改地址、投诉等问题。系统对高频售后问题使用规则快速回复，对复杂问题使用 RAG 检索本地售后知识库，再调用本地大模型生成答案。
```

## 2. 项目解决了什么问题

传统客服有几个痛点：

1. 高频问题重复，例如“退款多久到账”“怎么退货”“物流到哪了”。
2. 人工客服响应慢，用户等待时间长。
3. 售后政策和流程很多，人工容易漏说关键信息。
4. 普通大模型可能胡编订单状态，所以需要限制它只能基于知识库和示例订单回答。

本项目的解决方式：

1. 高频问题走规则回复，速度快。
2. 复杂问题走 RAG 知识库问答，覆盖面更广。
3. 订单查询走数据库中的结构化订单记录，不让大模型编订单。
4. 回复结构按照正式售后流程组织：确认诉求、核验订单、判断状态、处理方案、预计时效、下一步。
5. 支持微信公众号测试号，让项目不只是网页 Demo，也能模拟真实用户入口。

## 3. 整体架构怎么讲

可以这样讲：

```text
用户问题进入系统后，统一调用 chat() 方法。chat() 先调用 quick_answer() 判断是否命中高频售后规则。如果命中，就直接返回固定且规范的客服回复。如果没有命中，再判断是不是售后领域问题。非售后问题会交给本地大模型做简短闲聊；售后领域问题会先从 Chroma 向量数据库检索相关知识库内容，再把检索结果和用户问题一起发给 Ollama 本地大模型生成答案。
```

对应代码：

- `main.py`：客服核心逻辑和 Gradio 页面。
- `prepare_chroma.py`：把 `docs/` 文档生成向量库。
- `docs/`：售后知识库。
- `wechat_server.py`：微信公众号回调服务。
- `smoke_test.py`：自动化测试。

## 4. 什么是 RAG

RAG 是 Retrieval-Augmented Generation，中文一般叫“检索增强生成”。

简单讲：

```text
不是直接让大模型凭记忆回答，而是先从自己的知识库中检索出相关资料，再把资料交给大模型，让大模型基于资料回答。
```

这个项目里的 RAG 流程：

1. 把售后政策、售后流程、FAQ、订单物流资料放到 `docs/`。
2. 用 Embedding 模型把文档转成向量。
3. 向量保存到 Chroma。
4. 用户问问题时，也把用户问题转成向量。
5. 在 Chroma 里找语义最相近的文档。
6. 把检索到的文档内容拼到 Prompt 里。
7. 调用 Ollama 本地大模型生成最终回复。

面试时可以强调：

```text
RAG 的好处是可以让回答更贴近项目自己的业务知识库，并减少大模型胡编。比如售后政策、退款时效、退货流程都放在本地文档中，模型回答时会参考这些资料。
```

## 5. 为什么不用纯大模型回答

可以这样答：

```text
纯大模型回答有两个问题：第一是慢，每次都调用模型会增加响应时间；第二是容易幻觉，比如编造不存在的订单状态。电商售后里订单和退款属于比较敏感的信息，不能让模型随便猜。所以我把高频、确定性强的问题放到规则层处理，把复杂但需要知识库支持的问题交给 RAG。
```

本项目采用三层策略：

1. **规则层**：订单查询、退款流程、换货流程、投诉等高频问题。
2. **RAG 层**：规则没覆盖、但属于售后领域的问题。
3. **通用聊天层**：简单闲聊和非售后问题。

## 6. 为什么要做规则快速回复

规则快速回复适合以下问题：

- 答案固定。
- 高频出现。
- 对准确性要求高。
- 不需要复杂推理。

例如：

```text
退款多久到账？
退货退款怎么操作？
商品破损怎么办？
我要转人工。
```

好处：

1. 响应速度快，不需要等待大模型。
2. 结果稳定，每次回答一致。
3. 可以控制客服话术。
4. 可以避免大模型输出不符合业务规则的内容。

代码中主要对应：

- `quick_answer()`
- `is_standalone_service_intent()`
- `standalone_service_answer()`
- `order_contextual_answer()`

## 7. Chroma 向量数据库是什么

Chroma 是一个本地向量数据库。

可以这样讲：

```text
普通数据库通常按关键词、ID 或字段查询。向量数据库是按语义相似度查询。比如用户问“快递三天没动静怎么办”，知识库里写的是“物流超过 72 小时未更新可登记核查”，虽然文字不完全一样，但语义相近，向量数据库可以把这段资料检索出来。
```

项目中：

- `docs/` 是原始知识库。
- `prepare_chroma.py` 负责生成向量。
- `chroma/` 是生成后的本地向量库。
- `main.py` 中通过 `retriever.invoke(message)` 检索相关文档。

## 8. Embedding 是什么

Embedding 就是把文本变成一组数字向量。

可以这样理解：

```text
电脑本身不理解中文句子的含义，所以需要 Embedding 模型把句子转成向量。意思相近的句子，向量距离就更近。这样系统才能根据语义找资料，而不是只靠关键词匹配。
```

本项目使用：

```text
nomic-embed-text
```

它通过 Ollama 本地运行，用于把售后文档和用户问题转成向量。

## 9. Ollama 和 qwen3:4b 是做什么的

Ollama 是本地运行大模型的工具。

本项目用它运行：

```text
qwen3:4b
```

作用：

1. 对复杂问题生成自然语言回复。
2. 对简单非售后问题进行简短闲聊。
3. 在 RAG 流程中，根据检索出的售后知识库生成答案。

为什么选择本地模型：

1. 不依赖外部 API Key。
2. 本地演示方便。
3. 数据不需要发到第三方平台。
4. 适合新手学习大模型应用完整流程。

不足：

1. 本地模型效果受电脑性能影响。
2. 首次调用可能较慢。
3. 小模型复杂推理能力有限。

## 10. Prompt 是怎么设计的

本项目里有两类 Prompt：

### 售后客服系统 Prompt

要求模型：

- 始终用中文。
- 只处理电商售后相关问题。
- 不编造真实订单信息。
- 缺少订单号、凭证或核验信息时，要提醒用户补充。
- 回复按正式售后流程组织。

### 通用聊天 Prompt

要求模型：

- 可以简单聊天。
- 不能偏离售后客服身份。
- 遇到医疗、法律、金融等高风险问题，只做通用提醒，不给专业结论。

面试可以这样说：

```text
我用 Prompt 限制模型角色、回答范围和输出格式，避免模型输出推理过程、编造订单、或者回答和售后无关的内容。
```

## 11. 多轮上下文是怎么处理的

问题背景：

用户经常这样问：

```text
用户：订单 EC20260702002 物流到哪了？
用户：它还能改地址吗？
```

第二句话没有订单号，但“它”指的是上一单，所以系统需要从历史记录里找到订单号。

项目中的处理：

1. `extract_order_id()` 从当前输入中提取订单号。
2. 如果当前没有订单号，`should_use_history_order()` 判断是否应该继承历史订单。
3. 如果是“它、这个订单、状态、进度、改地址、退款”等追问，就从历史记录里提取订单号。
4. 如果只是“好的、知道了、嗯嗯”，就不继承订单，避免重复上一条结果。

这解决了之前的问题：

```text
用户：订单 EC20260702002 物流到哪了？
客服：返回物流信息
用户：好的
客服：不再重复物流信息，而是回复“好的，有其他售后问题可以继续发给小小易。”
```

## 12. 订单查询为什么不用大模型

订单查询优先读取项目数据库中的订单记录，历史版本中的 `ORDERS` 示例数据只作为早期演示思路。

可以这样答：

```text
订单状态属于结构化数据，不能让大模型自由生成。当前项目把订单、用户、工单、会话和凭证统一放到数据库层管理，查询订单时先从数据库读取状态、物流、退款等字段，再根据状态给出售后方案、核验要求和预计时效。
```

正式上线时可以继续对接：

- MySQL
- PostgreSQL
- MongoDB
- 电商平台订单 API
- 企业内部 OMS 系统

## 13. 微信公众号是怎么接入的

项目使用 `wechat_server.py` 提供 `/wechat` 接口。

微信接入流程：

```text
用户微信发消息
    ↓
微信服务器把消息推送到 /wechat
    ↓
FastAPI 解析 XML
    ↓
调用 main.py 的 chat()
    ↓
把回复包装成微信要求的 XML
    ↓
微信把消息发给用户
```

关键点：

1. 微信验证接口时，会传 `signature`、`timestamp`、`nonce`、`echostr`。
2. 服务端用 Token、timestamp、nonce 排序后做 SHA1 加密。
3. 加密结果和微信传来的 signature 一致，就返回 echostr，表示验证通过。
4. 用户发消息时，微信会 POST XML 到 `/wechat`。
5. 后端解析 XML 中的 `FromUserName`、`ToUserName`、`Content`。
6. 调用 `chat()` 得到回复，再返回 XML。

面试简答：

```text
我用 FastAPI 写了一个微信回调服务，先完成微信服务器签名校验，再解析用户 XML 消息，把文本内容传给客服主逻辑，最后按微信要求返回 XML 文本消息。
```

## 14. 为什么微信返回 XML

微信公众号被动回复接口要求返回 XML 格式。

本项目返回类似：

```xml
<xml>
  <ToUserName><![CDATA[用户]]></ToUserName>
  <FromUserName><![CDATA[公众号]]></FromUserName>
  <CreateTime>时间戳</CreateTime>
  <MsgType><![CDATA[text]]></MsgType>
  <Content><![CDATA[小小易的回复]]></Content>
</xml>
```

`CDATA` 的作用是保护中文、标点和特殊字符，避免 XML 解析出错。

## 15. 为什么要做测试

项目新增 `smoke_test.py`。

它的作用：

1. 快速验证核心功能是否还正常。
2. 避免修改代码后破坏已有功能。
3. 覆盖多轮上下文问题，比如“好的”不应该重复订单。
4. 检查模型回复中不能出现 `<think>`、提示词、角色设定等不该给用户看的内容。

面试可以这样说：

```text
我做的是冒烟测试，不是完整单元测试。它主要验证客服核心场景是否能跑通，比如订单查询、退款、换货、物流、投诉、闲聊和上下文承接。
```

## 16. 响应速度是怎么优化的

项目做了几层优化：

1. 高频售后问题优先走规则，不调用大模型。
2. 只有复杂售后问题才走 RAG。
3. 非售后简单聊天才走通用模型。
4. Ollama 调用时设置 `keep_alive="30m"`，让模型保持一段时间热启动。
5. 限制模型输出长度，例如 `num_predict`，避免回复过长导致变慢。
6. 对模型输出做清洗，去掉 `<think>` 或提示词泄露内容。

可以这样回答：

```text
我没有把所有请求都交给大模型，而是先用规则判断高频问题。大部分售后问题其实是标准流程，规则回复速度更快。只有规则覆盖不到的问题才走知识库检索和大模型生成。
```

## 17. 项目目前的不足

面试时主动讲不足，会显得你理解项目边界。

当前不足：

1. 当前订单、工单和会话使用本地 SQLite 演示数据库，正式上线还需要对接企业真实订单中心、物流系统、退款系统和发票系统。
2. 后台管理页面是轻量演示版，正式客服工作台还需要工单分配、处理备注、状态流转、权限控制和 SLA 提醒。
3. 图片凭证已经支持上传和记录，但还没有接 OCR 或多模态识别，暂时不能自动判断破损、面单或商品信息。
4. RAG 知识库文档数量较少，覆盖范围还可以继续扩展。
5. 本地模型效果受机器性能和模型大小影响，DeepSeek 等云端模型也需要考虑费用、限流和稳定性。

## 20. 面试常见问题和参考回答

### 问：你这个项目的核心技术点是什么？

答：

```text
核心是规则快速回复和 RAG 检索增强生成的结合。高频售后问题直接走规则，保证速度和稳定性；复杂问题先从 Chroma 知识库检索相关资料，再让本地大模型基于资料生成答案。这样既能提高响应速度，也能减少大模型胡编。
```

### 问：为什么使用 RAG？

答：

```text
因为售后客服需要回答公司自己的政策和流程，纯大模型不一定知道，而且可能编造。RAG 可以先检索本地售后知识库，再让模型基于检索内容回答，更适合企业知识问答场景。
```

### 问：为什么有了大模型还要写规则？

答：

```text
售后高频问题很多答案是固定的，比如退款时效、退货流程、转人工、订单查询。规则回复更快、更稳定，也更容易控制话术。大模型主要用于规则没覆盖的复杂问题。
```

### 问：如果用户问订单物流，你怎么避免模型乱编？

答：

```text
订单物流不交给大模型自由回答，而是先用正则提取订单号，再从数据库订单表中查询订单状态、物流单号和最近轨迹。正式项目中这里可以继续替换为企业订单中心或物流接口。
```

### 问：多轮对话怎么处理？

答：

```text
我会先判断当前输入里有没有订单号。如果没有，再判断用户是不是在追问上一单，比如“它现在什么状态”“这个订单能改地址吗”。如果是追问，就从历史消息里提取订单号；如果只是“好的”“知道了”，就不继承订单，避免重复回复。
```

### 问：微信公众号怎么接入？

答：

```text
我用 FastAPI 提供 /wechat 接口。微信测试号配置 URL 和 Token 后，会先发 GET 请求做签名校验。校验通过后，用户发消息时微信会 POST XML 到接口。我解析 XML 里的文本内容，调用 chat() 得到回复，再按微信要求返回 XML。
```

### 问：这个项目怎么保证回答质量？

答：

```text
主要从四方面保证：第一，高频问题走规则，输出稳定；第二，复杂问题走知识库检索，减少胡编；第三，Prompt 限制模型角色、范围和输出格式；第四，smoke_test.py 覆盖核心售后场景，防止修改后出现明显回归。
```

### 问：如果知识库更新了怎么办？

答：

```text
修改 docs/ 里的售后资料后，需要运行 prepare_chroma.py 重新生成 Chroma 向量索引。这样新资料才会参与后续检索。
```

## 21. 你要重点背下来的 8 句话

1. 这个项目是电商售后智能客服，核心是规则快速回复 + RAG 知识库问答。
2. 高频问题走规则，复杂问题走 Chroma 检索和本地大模型生成。
3. RAG 是先检索知识库，再让模型基于资料回答，可以减少胡编。
4. Chroma 是向量数据库，用来按语义相似度检索售后文档。
5. Embedding 是把文本转成向量，意思相近的句子向量距离更近。
6. 订单状态不用大模型编，而是从数据库订单记录查；正式项目应继续对接真实订单中心和物流接口。
7. 微信接入是 FastAPI 接收 XML，调用 chat()，再返回 XML 给微信。
8. 我做了冒烟测试，覆盖订单、物流、退款、换货、投诉、闲聊和上下文承接。

## 22. 自动学习机制补充

这次新增的学习机制可以这样理解：

```text
系统先用规则和知识库回答用户问题。如果没有命中规则，需要调用 RAG 或通用大模型生成答案，就把这组“问题-答案”记录下来。记录会保存到 Excel；如果配置了 MySQL，也会同步写入数据库。同时，系统会把大模型兜底生成的答案写入本地学习缓存。下次用户问完全相同的问题时，系统会优先从学习缓存里回答，减少重复调用大模型。
```

相关文件：

- `learning_store.py`：负责问答记录、Excel 写入、MySQL 写入和学习缓存。
- `data/qa_learning_records.xlsx`：自动生成的问答 Excel 记录。
- `data/learned_qa.jsonl`：本地学习缓存，用于复用已学习的问题答案。

Excel、MySQL、学习缓存的区别：

```text
Excel：方便导出和人工查看，适合演示和审核。
MySQL：适合正式项目做结构化存储、后台查询和统计。
学习缓存：适合系统运行时快速复用答案，减少重复调用大模型。
```

面试时可以这样说：

```text
我没有把所有问题都放进内存里，而是落盘保存到 Excel 和可选的 MySQL。大模型兜底回答会进入本地学习缓存，后续相同问题可以直接复用。如果希望学会后清理临时数据，可以设置 LEARNING_DELETE_AFTER_LEARNED=1，这样系统会删除 Excel/MySQL 中已学习的临时记录，只保留轻量缓存。
```

需要注意：

```text
正式生产环境不能让系统无审核地学习所有大模型答案。因为大模型可能答错，如果错误答案直接进入知识库，以后就会稳定地重复错误。更稳的方案是：先把新问答保存为待审核记录，人工确认后再写入正式知识库，并重新生成向量索引。
```

MySQL 表设计：

```text
表名：qa_learning_records

id：自增主键
question_hash：问题归一化后的哈希值
question：用户问题
answer：客服回答
source：回答来源，例如 quick_answer、rag_answer、general_model_chat、learned_answer
status：记录状态，例如 recorded、learned
created_at：创建时间
learned_at：学习时间
```

如果面试官问“为什么要用 question_hash”，可以回答：

```text
因为用户可能重复问同一个问题，直接用长文本查询不方便。先把问题做归一化，再生成哈希，可以更快判断这个问题是否已经学过。
```

新增后你要额外记住这 3 句话：

1. 问答学习模块会把用户问题和客服回答保存到 Excel，配置后也能同步写入 MySQL。
2. 大模型兜底答案会进入本地学习缓存，后续相同问题可以直接复用。
3. 正式上线不能无审核自动学习，应该先人工审核，再写入正式知识库。

这次把项目从“聊天演示”继续升级成更接近真实售后系统的版本。

新增模块：

```text
app_database.py：用户、订单、工单、会话、图片凭证的数据层。
```

### 为什么要加数据库层

可以这样回答：

```text
之前订单数据是 Python 字典，只适合演示。真实售后系统里，订单、用户、工单、会话和凭证都属于结构化业务数据，应该放到数据库里统一管理。所以我加了 app_database.py，把数据访问集中到一个模块。默认用 SQLite 方便本地运行，配置 APP_DB_BACKEND=mysql 后可以切到 MySQL。
```

当前数据表包括：

```text
users：用户登录信息
orders：订单信息
tickets：售后工单
conversations：网页和微信会话记录
evidence_files：图片凭证记录
```

### 工单系统怎么设计

工单用于记录需要人工跟进的问题，例如：

- 投诉升级
- 转人工
- 物流核查
- 退款进度
- 破损、错发、少件凭证审核

可以这样说：

```text
当用户表达投诉、转人工、物流核查、退款进度或上传凭证时，系统会把问题登记为工单。后台管理页面可以查看工单类型、状态、优先级、处理进度和更新时间。正式项目里可以继续给工单加客服分配、状态流转和处理备注。
```


### 图片上传为什么重要

售后场景里，破损、错发、少件、质量问题通常需要证据。

可以这样回答：

```text
我支持用户上传图片凭证，并把文件保存路径、用户、订单号和工单号记录到数据库。当前版本主要做凭证保存和后台查看，后续可以接 OCR 或多模态模型，自动判断图片里是否有破损、面单或商品信息。
```

### DeepSeek 怎么接入

项目新增了 `model_provider.py`，统一封装大模型调用。

配置方式：

```text
DEEPSEEK_API_KEY：DeepSeek API Key
LLM_PROVIDER=auto：有 Key 时优先 DeepSeek，失败时回退 Ollama
LLM_PROVIDER=ollama：只用本地 Ollama
LLM_PROVIDER=deepseek_strict：只用 DeepSeek，失败就报错
```

可以这样讲：

```text
我把大模型调用抽成 model_provider.py，不让业务代码直接依赖某一个模型。这样本地演示时可以用 Ollama，有 DeepSeek 余额时可以优先走 DeepSeek，后续要换 OpenAI、通义千问或其他模型也比较方便。
```

### 微信历史为什么要持久化

之前微信历史保存在内存里，服务重启后上下文会丢失。

升级后：

```text
微信 openid 会作为 session_id 的一部分。
用户消息和客服回复都会写入 conversations 表。
下次用户继续发消息时，系统从数据库读取最近 12 条记录作为上下文。
```

面试可以这样答：

```text
微信端不能只用内存保存上下文，因为服务重启或多进程部署都会丢状态。我把微信会话写入数据库，按 openid 读取最近消息，这样上下文更稳定，也方便后台客服查看历史记录。
```
